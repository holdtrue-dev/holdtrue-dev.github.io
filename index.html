#what it is

holdtrue checks that code does what you meant by checking it against a contract you wrote and approved, instead of by reading the code.

You describe what you want in plain language. holdtrue turns that into a contract: type signatures, properties, and a spec a solver can check. You review the contract. Then an implementation is written, and holdtrue runs the contract against it. The output is not a diff. It is a report that says, for each thing you asked for, whether it is proven, only sampled, or broken.

The point is to move your attention from "is this code correct?" to "does this contract say what I meant?". For small, well-understood functions that is an easier question. It is also the question every reviewer is really trying to answer when they read a diff.

#the idea

Code review asks people to read an implementation and guess whether it matches intent. That is slow, it misses things, and it does not scale. The guarantee you actually want is that behaviour matches intent. Reading code only gives you a weak, manual approximation of it.

holdtrue changes the order. Intent is written down first and turned into checks before any code exists. The checks come from intent, not from an implementation, so they are not quietly shaped to fit whatever the code happens to do. A human reviews and owns the contract. The implementation serves the contract, not the other way around.

When holdtrue says GUARANTEED, it means a solver proved the implementation satisfies the contract for every input in the declared domain, and that the contract is strong enough to catch injected faults. It does not mean the contract matches what you meant. That part is still yours. holdtrue makes the gap visible instead of hiding it.

#how it works

The loop has a few steps.

  • Intent. You write what you want in plain language. holdtrue asks about the parts usually left implicit: edge cases, error behaviour, what counts as valid input.
  • Contract. The answers become a machine-checkable contract: types, properties over generated inputs, and a spec the symbolic checker can prove. You read and approve it. You can edit it. It is the source of truth.
  • Implementation. Written separately, against the approved contract. The contract author and the code author are kept apart, so the code cannot be fitted to the contract author's private reasoning.
  • Verification. holdtrue runs the whole contract: types, the symbolic proof, property tests (including ones the implementer never saw), a probe that feeds the contract deliberately broken code to check it gets rejected, and mutation testing that injects faults to check the contract has teeth.
  • Report. A verdict for each piece of intent.

the three verdicts

  • GUARANTEED. Proven over the whole input domain, and the contract caught injected bugs and rejected broken stand-ins. You do not need to read the code.
  • UNGUARANTEED. The evidence is only sampled or partial. This still needs human review, and holdtrue says so plainly rather than showing green.
  • FAILED. A concrete input breaks the contract. The report gives you that input.

Nothing is weakened to make a build pass. If a check can only be satisfied by changing the contract, that change goes back to you for approval and is recorded.

$getting started

holdtrue targets Python pure functions for now. You need Python 3.12 and uv.

git clone https://github.com/holdtrue-dev/holdtrue
cd holdtrue
uv sync
uv run holdtrue verify examples/clamp \
  --impl examples/clamp/controls/correct.py

That runs the full contract for a clamp function against a correct implementation. You see each check, then the verdict: GUARANTEED, backed by a symbolic proof, mutation testing, and the negative probe.

To see the other outcomes:

# a bug: the report hands you a counterexample
uv run holdtrue verify examples/clamp \
  --impl examples/clamp/controls/buggy.py

# a correct function, too-weak contract: refused a guarantee
uv run holdtrue verify examples/clamp \
  --impl examples/clamp/controls/correct.py \
  --manifest contract/manifest_weak.yaml

The last one is the one to watch. The function is correct and the proof passes, but holdtrue still refuses to say GUARANTEED, because the contract is weak enough to also accept broken code. A green check there would not mean anything, so it does not give you one.

#what it proves, and what it doesn't

holdtrue can prove a lot for a pure, fully typed function over a domain the checker can exhaust: bounded integers, finite enums, and the like. For those it proves the implementation matches the written contract for every input, confirms the contract is not vacuous, and confirms it rejects broken implementations.

It cannot prove that the contract matches your intent. If you forget to specify a case, the proof still passes and the report still says GUARANTEED for what you did specify. holdtrue lists which parts of the intent were interrogated so the gaps are visible, but closing them is human judgment.

It also cannot say much about code it cannot reason over symbolically: floating point, unbounded loops, time, IO, randomness. Those come back as UNGUARANTEED, which is an honest result, not a failure. The line between "proven now" and "still needs a human" is the whole point, and the tool reports it straight.

#status

Early. One language, one narrow domain, proven end to end. The contract author and the implementer run as separate steps today; running them as separate model contexts is the next step. After that, the plan is to widen past pure functions.
